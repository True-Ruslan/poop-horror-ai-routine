# DECISIONS.md

Журнал архитектурных и продуктовых решений.

## ADR-0001 — Godot 4.7+

Дата: 2026-07-07

Решение: проект разрабатывается на Godot 4.7+ и GDScript.

Причина:

- низкий порог входа;
- текстовый формат сцен удобен для review;
- проект проще развивать маленькими итерациями.

## ADR-0002 — Маленькие scripted events

Дата: 2026-07-07

Решение: хоррор прототипа строится на маленьких scripted events, а не на сложном AI-враге.

Причина:

- меньше багов;
- проще тестировать;
- соответствует жанру бытового психологического хоррора.

## ADR-0003 — Интерактивные объекты как отдельные сцены

Дата: 2026-07-07

Решение: дверь, записка, выключатель, монитор и другие предметы оформляются как отдельные сцены.

Причина:

- переиспользование;
- удобство настройки;
- маленькие reviewable changes.

## ADR-0004 — Проект становится конкретной игрой

Дата: 2026-07-07

Решение: репозиторий развивается как `AI Routine: Last Commit`, а не только как общий шаблон.

Причина:

- конкретный сценарий помогает двигаться быстрее;
- квартира разработчика даёт понятный playable slice.

## ADR-0005 — Первая локация: квартира разработчика

Дата: 2026-07-07

Решение: `Main.tscn` запускает `DeveloperApartment.tscn`.

Причина:

- локация соответствует теме рабочей рутины;
- в ней достаточно объектов для первого prototype.

## ADR-0006 — Сначала собственная геометрия и простые лицензии

Дата: 2026-07-07

Решение: prototype использует примитивы Godot, внешние ресурсы добавляются только с понятными лицензиями.

Причина:

- контроль прав;
- безопасная будущая публикация;
- обязательный `docs/CREDITS.md`.

## ADR-0007 — Осознанная сверка с upstream-шаблоном

Дата: 2026-07-07

Решение: изменения из `True-Ruslan/godot-simple-tamplate` не переносятся автоматически.

Причина:

- игра уже отличается от generic template;
- перенос должен быть reviewable и безопасным.

## ADR-0008 — Production scope: 45–60 минут

Дата: 2026-07-12

Решение: целевая 1.0 — законченная игра на 45–60 минут с прологом, тремя актами и четырьмя финалами.

Финалы:

- `MERGE`;
- `REVERT`;
- `BLACKOUT`;
- `CLEAN BUILD`.

Причина:

- масштаб достаточен для самостоятельного коммерческого продукта;
- одна глубокая локация реалистичнее нескольких поверхностных;
- альтернативные финалы увеличивают ценность без неконтролируемого расширения мира.

## ADR-0009 — Одна production-квартира вместо большой карты

Дата: 2026-07-12

Решение: основной игровой мир — квартира 38–45 м² с прихожей, коридором, комнатой, кухней и санузлом.

Причина:

- игрок запоминает нормальную планировку;
- пространственные изменения становятся заметными;
- высокая плотность деталей важнее масштаба;
- проще выдержать production quality и performance budget.

## ADR-0010 — Grounded stylized realism

Дата: 2026-07-12

Решение: финальный стиль — grounded stylized realism с low/mid-poly моделями и PBR-материалами. PSX/VHS-эффекты используются только в сюжетных нарушениях.

Причина:

- чистый greybox не подходит для публикации;
- постоянный VHS-фильтр скрывал бы качество и ухудшал доступность;
- стилизованный реализм достижим для небольшой команды и сохраняет бытовую правдоподобность.

## ADR-0011 — Data-driven narrative

Дата: 2026-07-12

Решение: progression, objectives, event sequences, dialogue и endings хранятся в Godot Resources, а не в объектных скриптах.

Причина:

- текущая цепочка строковых event ID плохо масштабируется;
- данные можно валидировать и тестировать;
- локализация и save recovery становятся предсказуемыми;
- интерактивные prefab остаются переиспользуемыми.

## ADR-0012 — NarrativeDirector и EventSequenceRunner

Дата: 2026-07-12

Решение: narrative state управляется `NarrativeDirector`, а асинхронные постановочные события выполняет `EventSequenceRunner`.

Причина:

- текущий `HorrorEventManager` сообщает о завершении сразу после callback;
- production events требуют ожидания, параллельных действий, отмены и восстановления;
- объекты не должны напрямую связывать следующие сюжетные шаги.

## ADR-0013 — Component-based interactive actors

Дата: 2026-07-12

Решение: новые интерактивные объекты строятся из отдельных interaction, visual, audio, highlight и saveable components.

Причина:

- единый interaction contract;
- меньше дублирования;
- корректное сохранение состояний;
- отдельное тестирование поведения;
- возможность постепенно мигрировать prototype objects.

## ADR-0014 — Autosave и checkpoints без ручных сохранений

Дата: 2026-07-12

Решение: игра использует checkpoints, autosave и backup; ручное save menu не реализуется.

Причина:

- игра короткая;
- произвольное сохранение внутри event sequence создаёт много некорректных состояний;
- checkpoint перед финалом позволяет открыть альтернативные endings без полного replay.

## ADR-0015 — Минимальный HUD и отдельные focus modes

Дата: 2026-07-12

Решение: постоянный HUD показывает только crosshair и interaction prompt. Терминал, телефон и inspect используют отдельные modes.

Причина:

- постоянные панели мешают атмосфере;
- terminal mode должен позволять событиям происходить в комнате;
- phone mode становится самостоятельным narrative channel;
- accessibility можно реализовать без перегрузки основного экрана.

## ADR-0016 — Production rebuild через legacy adapters

Дата: 2026-07-12

Решение: старые менеджеры и playable route не переписываются одним PR. Новая архитектура сначала загружает prototype через adapters.

Причина:

- `main` должен оставаться запускаемым;
- regression route позволяет сравнивать поведение;
- постепенная миграция снижает риск полной поломки проекта.

## ADR-0017 — RU/EN и accessibility входят в 1.0

Дата: 2026-07-12

Решение: русский, английский и согласованный набор accessibility-настроек являются обязательной частью первого публичного релиза.

Причина:

- текст и звук критичны для понимания сюжета;
- эффекты камеры и вспышки требуют пользовательского контроля;
- добавление localization/accessibility в конце обычно приводит к дорогой переработке UI и данных.

## ADR-0018 — CI, narrative validation и release channels

Дата: 2026-07-12

Решение: проект использует автоматические проверки, narrative validator и каналы Development, Playtest, Release.

Причина:

- progression logic нельзя проверять только вручную;
- debug-функции не должны попадать в публичную сборку;
- exported build должен проверяться отдельно от редактора;
- лицензии, localization keys и resource paths требуют автоматической проверки.
