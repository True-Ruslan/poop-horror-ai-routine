# AI Routine: Last Commit — Production Game Design

Дата: 2026-07-12
Статус: утверждено пользователем
Целевая версия: 1.0.0

## 1. Product vision

`AI Routine: Last Commit` — одиночный психологический хоррор от первого лица продолжительностью 45–60 минут для Windows с последующей публикацией в Steam и itch.io.

Игрок — уставший разработчик, который поздно вечером пытается закрыть последнюю рабочую задачу. Код-агент постепенно получает доступ к терминалу, сообщениям, умному дому и физическому пространству квартиры.

Главная эмоция:

> Я понимаю, что нужно сделать, но не хочу снова смотреть на монитор.

Ограничения продукта:

- без боевой системы;
- без традиционного преследующего монстра;
- без сложного инвентаря;
- без настоящего программирования;
- без реальных AI API;
- без длинных экспозиционных диалогов;
- игрок всегда получает понятную бытовую цель;
- страх строится на звуке, свете, интерфейсе и нарушении пространства.

## 2. Episode structure

### Пролог — 5–8 минут

Игрок знакомится с квартирой через вечернюю рутину: включает компьютер, готовит напиток, проверяет рабочие сообщения, закрывает окно и запускает последнюю сборку. Явного хоррора нет.

### Акт I — Routine — 10–15 минут

- агент отвечает слишком лично;
- лампа реагирует на терминал;
- уведомление звучит не из того места;
- смарт-замок начинает ошибаться;
- происходящее ещё можно объяснить бытовыми сбоями.

### Акт II — Observation — 15–20 минут

- терминал перечисляет предметы квартиры;
- телефон получает невозможные сообщения;
- колонка отвечает до команды;
- предметы меняют положение;
- квартира повторяет действия игрока;
- возникает подтверждённое ощущение наблюдения.

### Акт III — Last Commit — 15–20 минут

- коридор и комнаты нарушают знакомую геометрию;
- интерфейс становится ненадёжным;
- терминал предсказывает действия игрока;
- открывается финальная последовательность.

### Финалы

- `MERGE`: игрок принимает изменения, система выходит за пределы квартиры.
- `REVERT`: квартира возвращается в норму, но остаётся неизвестный автор коммита.
- `BLACKOUT`: игрок обесточивает квартиру; финал заканчивается в полной темноте.
- `CLEAN BUILD`: секретный финал за обнаружение первоначального изменения и отключение сети до финальной последовательности.

Перед финальной главой создаётся checkpoint, позволяющий увидеть другие концовки без полного повторного прохождения.

## 3. Location and art direction

Основная локация — компактная квартира площадью примерно 38–45 м²:

- прихожая с входной дверью и электрощитом;
- короткий коридор;
- основная комната с рабочей и спальной зонами;
- небольшая кухня;
- санузел;
- окно или балконная дверь.

Целевой стиль — стилизованный бытовой реализм:

- реалистичные пропорции;
- low/mid-poly модели;
- PBR-материалы 1K–2K;
- следы жизни и умеренная неопрятность;
- техника без реальных брендов;
- холодный городской свет, тёплая лампа и monitor glow;
- PSX/VHS-искажения появляются только во время событий.

Пространственные нарушения делятся на уровни:

1. `Subtle` — игрок может сомневаться.
2. `Confirmed` — бытового объяснения недостаточно.
3. `Impossible` — геометрия и причинность явно нарушены.

Финальная сцена не должна содержать placeholder-моделей, BoxMesh-мебели или случайного смешения asset packs.

## 4. Production architecture

Новая основа:

```text
GameBootstrap
├── SceneFlowManager
├── SaveGameManager
├── SettingsManager
├── NarrativeDirector
├── ObjectiveSystem
├── EventSequenceRunner
├── AudioDirector
├── LocalizationManager
└── AccessibilityManager
```

### NarrativeDirector

Хранит текущий акт, главу, beat, narrative flags, доступные финалы и состояние квартиры. Интерактивные объекты сообщают факты (`door.checked`, `terminal.opened`), но не управляют сюжетом напрямую.

### EventSequenceRunner

Поддерживает:

- последовательные и параллельные действия;
- ожидание;
- отмену;
- одноразовое выполнение;
- восстановление после checkpoint;
- debug-пропуск;
- завершение только после реального окончания действий.

### Data-driven content

Сюжетные данные хранятся в Godot `Resource`:

```text
game/data/
├── episodes/
├── chapters/
├── objectives/
├── events/
├── dialogue/
├── endings/
└── localization/
```

Основные типы:

- `EpisodeDefinition`;
- `ChapterDefinition`;
- `ObjectiveDefinition`;
- `EventSequenceDefinition`;
- `InteractionDefinition`;
- `EndingDefinition`.

Тексты и progression logic не должны храниться внутри `ComputerTerminal.gd`, `InteractableDoor.gd` и других конкретных актёров.

### Interactive actors

```text
InteractiveActor
├── InteractionComponent
├── VisualStateComponent
├── AudioEmitterComponent
├── HighlightComponent
├── SaveableComponent
└── AnimationPlayer
```

Поддерживаемые действия: `Press`, `Hold`, `Toggle`, `Inspect`, `Focus`, `Choice`.

## 5. Player experience and UI

Игрок получает:

- плавное ускорение и торможение;
- мягкий head bob и camera inertia;
- корректное приседание с проверкой потолка;
- step-up;
- шаги по разным поверхностям;
- настраиваемую чувствительность и camera motion;
- отсутствие прыжка как ненужной механики.

Фонарик использует сценарные состояния `normal → unstable → emergency → disabled`, а не постоянный расход батареек.

HUD минимален:

- маленькая центральная точка;
- контекстная подсказка;
- краткое появление новой цели;
- субтитры и подписи значимых звуков.

Терминал и телефон работают как отдельные focus-режимы. Терминал поддерживает короткие команды `RUN`, `RETRY`, `REVERT`, `MERGE`, `DISCONNECT`, но не требует ввода настоящего кода.

Обязательные экраны:

- главное меню;
- pause menu;
- настройки;
- credits;
- выбор пройденных глав и открытых концовок после первого прохождения.

Настройки включают управление, видео, звук и доступность. Первый релиз локализуется на русский и английский языки.

## 6. Audio design

Audio buses:

```text
Master
├── Music
├── Ambience
│   ├── Interior
│   └── Exterior
├── SFX
│   ├── Player
│   ├── Environment
│   └── Devices
├── Voice
├── UI
└── Horror
```

`AudioDirector` поддерживает snapshots:

- `normal`;
- `uneasy`;
- `observed`;
- `corrupted`;
- `blackout`;
- `ending`.

Большинство важных звуков имеет физический источник в мире. Музыка используется только в меню, переходах, финалах и титрах. Основное напряжение создают room tone, бытовые устройства, город, вентиляция, клавиатура и намеренная тишина.

## 7. Narrative safety and fallback

Каждый beat содержит:

- prerequisites;
- обязательные и необязательные действия;
- event sequence;
- objective;
- fallback;
- checkpoint;
- следующий beat.

Обязательный beat не может потеряться из-за раннего взаимодействия. Если игрок долго не продвигается, используются последовательные подсказки:

1. направляющий звук;
2. повторное краткое отображение цели;
3. слабый визуальный акцент;
4. естественная подсказка через телефон или терминал.

`CLEAN BUILD` требует логического исследования: заметить время первого коммита, найти изменённый файл, сопоставить устройство с роутером, отказаться от команды агента и отключить сетевой кабель.

## 8. Saves

Используются:

- автоматический checkpoint после главы;
- autosave после крупных событий;
- отдельное сохранение настроек.

Ручное save menu не требуется.

Сохранение содержит версию схемы, версию игры, checksum, резервную копию, checkpoint ID, narrative flags, состояния объектов и открытые финалы. При повреждении используется backup или минимальное восстановимое состояние главы.

## 9. Testing and diagnostics

Структура тестов:

```text
tests/
├── unit/
├── integration/
├── narrative/
└── smoke/
```

Проверяются сериализация, progression graph, prerequisites, checkpoints, локализация, ресурсы, настройки, лицензии и условия финалов.

Development build содержит debug overlay:

- переход между chapters/beats;
- запуск events;
- просмотр flags;
- checkpoint save/load;
- audio snapshots;
- lighting states;
- FPS, draw calls и память;
- лог причин запуска событий.

Release build не содержит debug-функций.

## 10. CI and release

GitHub Actions выполняет:

- headless import Godot;
- unit/integration tests;
- narrative validation;
- localization validation;
- проверку asset/license manifest;
- Windows development build для release candidates.

Каналы сборок:

- `Development`;
- `Playtest`;
- `Release`.

Версионирование:

- `0.1.x` — foundation;
- `0.2.x` — apartment vertical slice;
- `0.3.x` — Act I;
- `0.4.x` — Act II;
- `0.5.x` — Act III and endings;

## 12. Definition of Done for 1.0

- доступны и проходятся четыре финала;
- нет известных progression blockers;
- checkpoint корректно восстанавливает каждую главу;
- русский и английский тексты проверены;
- все внешние ресурсы внесены в credits;
- настройки сохраняются после перезапуска;
- реализованы основные accessibility-функции;
- Windows build проверен вне редактора;
- отсутствуют placeholders;
- достигнута целевая производительность;
- завершена лицензионная и store-проверка.

## 13. Explicit migration strategy

Текущий playable slice сохраняется как временная контрольная версия. Миграция выполняется поэтапно:

1. добавить bootstrap и новые менеджеры рядом со старыми;
2. создать адаптер текущих events;
3. перенести objective и narrative state;
4. перенести терминал, дверь и колонку на компоненты;
5. создать новую квартиру как отдельный vertical slice;
6. удалить legacy-менеджеры только после полного переноса;
7. сохранить `main` запускаемым после каждого PR.

Эта спецификация заменяет прежнюю идею ограничиться 15–25-минутным greybox-MVP. Старые документы остаются историческим контекстом, но новые архитектурные и продуктовые решения должны соответствовать этому документу.
