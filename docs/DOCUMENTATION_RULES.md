# DOCUMENTATION_RULES.md

## Цель

Документация проекта должна позволять любому код-агенту быстро понять:

- что это за игра;
- что уже сделано;
- что находится в работе;
- что нужно делать дальше;
- какие решения уже приняты;
- какие файлы нельзя менять без причины.

## Источники правды

| Документ | За что отвечает | Когда обновлять |
|---|---|---|
| `README.md` | Краткий вход в проект | Когда меняется назначение проекта, запуск или структура |
| `AGENTS.md` | Правила работы код-агента | Когда меняется процесс разработки |
| `docs/PROJECT_STATE.md` | Текущее состояние проекта | После каждой завершённой итерации |
| `docs/TASKS.md` | План задач и backlog | При начале, завершении или изменении задачи |
| `docs/SCENARIO.md` | Сюжет, главы, цели, события | При изменении сценария, целей или интерактивных объектов |
| `docs/GAME_DESIGN.md` | Дизайн-правила и ограничения | При изменении игровых принципов |
| `docs/DECISIONS.md` | Архитектурные и продуктовые решения | При добавлении систем, зависимостей или изменении подхода |
| `docs/CHANGELOG.md` | История изменений | В каждом PR с пользовательски заметными изменениями |
| `docs/CREDITS.md` | Ассеты, авторы, лицензии | При добавлении любого внешнего ассета |
| `docs/ASSET_SOURCES.md` | Правила выбора ассетов | При изменении лицензионной политики |
| `docs/ROADMAP.md` | Направление разработки | При изменении ближайших этапов |

## Обязательное правило для каждого PR

Если PR меняет код, сцену, механику, сюжет, структуру проекта или ассеты, он должен обновить минимум один документ.

Минимальный набор:

- `docs/TASKS.md` — отметить, что сделано и что дальше;
- `docs/CHANGELOG.md` — описать изменение;
- `docs/PROJECT_STATE.md` — обновить текущее состояние.

Дополнительно:

- меняется сценарий → обновить `docs/SCENARIO.md`;
- меняется архитектура → обновить `docs/DECISIONS.md`;
- добавлен ассет → обновить `docs/CREDITS.md`;
- меняется управление или запуск → обновить `README.md`.

## Формат задач

Каждая итерация в `docs/TASKS.md` должна иметь:

```md
## Iteration N — Название

Цель: один короткий абзац.

- [ ] Конкретная задача 1.
- [ ] Конкретная задача 2.

Definition of Done:

- Проверяемый критерий 1.
- Проверяемый критерий 2.
```

## Формат changelog

Использовать разделы:

```md
## [0.x.0] — YYYY-MM-DD

### Added

- ...

### Changed

- ...

### Fixed

- ...

### Removed

- ...
```

Не добавлять пустые разделы.

## Формат решений

Каждое решение в `docs/DECISIONS.md` должно иметь:

```md
## ADR-000X — Название

Дата: YYYY-MM-DD

Решение: коротко.

Причина:

- причина 1;
- причина 2.
```

## Что нельзя делать с документацией

- Не удалять историю решений без отдельного объяснения.
- Не заменять конкретные задачи общими словами.
- Не писать `todo later` без описания следующего шага.
- Не оставлять несоответствие между `TASKS`, `CHANGELOG` и фактическими файлами.
- Не добавлять ассеты без записи в `CREDITS`.
