# NARRATIVE_DESIGN.md

## Назначение

Этот документ описывает, за счёт чего игра должна пугать и запоминаться.

Ключевая мысль: игра держится не на большой графике, а на мелких нарушениях привычной рутины. Игрок должен узнавать обычную рабочую ситуацию, а затем замечать, что она стала чуть неправильной.

## Главная формула вайба

```text
обычная рабочая рутина + цифровая уверенность + бытовая уязвимость + звук, который пришёл не оттуда
```

Игрок не должен думать: «здесь поставили скример».

Игрок должен думать:

```text
я же точно не нажимал это
```

## Опорные столпы

### 1. Рабочая достоверность

Ситуации должны выглядеть знакомо для разработчика:

- поздняя задача;
- автотесты;
- терминал;
- лог сборки;
- TODO-заметки;
- странно уверенный код-агент;
- усталость от мелкого бага.

Важно: не делать симулятор программирования. Игрок не пишет реальный код. Он нажимает понятные бытовые действия: открыть терминал, проверить лампу, закрыть окно, отключить питание.

### 2. Маленькое нарушение

Каждая страшная сцена начинается с очень маленького сбоя:

- строка в логе появилась на секунду раньше, чем игрок нажал клавишу;
- лампа мигнула не в тот момент;
- звук уведомления прозвучал из комнаты, а не из компьютера;
- текст задачи изменился на одно слово;
- курсор печатает сам, но слишком медленно.

### 3. Контроль постепенно исчезает

У игрока сначала есть иллюзия контроля:

- можно включить и выключить свет;
- можно закрыть сообщение;
- можно отвернуться от монитора;
- можно подойти к двери.

Затем каждое действие начинает возвращаться против него:

- свет включается сам;
- сообщение закрывается, но появляется новое;
- монитор продолжает подсвечивать комнату;
- дверь отвечает текстом, как будто это не дверь, а чат.

### 4. Звук важнее картинки

Главный страх должен идти через звук:

- короткий щелчок реле;
- тихий электрический писк;
- клавиатурный ввод без рук;
- низкий комнатный гул;
- короткое уведомление, которое звучит слишком близко;
- голос/синтез речи, который произносит не ту фразу.

## Темп первого эпизода

| Фаза | Примерная длительность | Состояние игрока | Главный инструмент |
|---|---:|---|---|
| Вход в рутину | 2–4 мин | всё нормально | бытовые задачи |
| Первый сбой | 3–5 мин | странно, но объяснимо | терминал + лампа |
| Контур квартиры | 4–6 мин | дом связан с кодом | дверь + колонка |
| Потеря контроля | 4–6 мин | нельзя доверять интерфейсу | HUD + монитор |
| Финальный выбор | 2–4 мин | нужно решить, чем жертвовать | merge/revert/force close |

## Сценарные классы хоррор-моментов

### Terminal dread

Игрок смотрит на текстовый интерфейс, где всё выглядит технически нормально, но смысл становится личным.

Примеры:

- `Tests passed: 0 failed. 1 observer attached.`
- `Agent: я уже проверил дверь.`
- `Warning: user did not read generated code.`
- `Build completed before command execution.`

### Room response

Комната отвечает на событие в компьютере.

Примеры:

- после строки в терминале щёлкает лампа;
- после ошибки сборки моргает смарт-замок;
- после закрытия окна звучит уведомление из колонки;
- после отката коммита в квартире становится тише, чем должно быть.

### Interface betrayal

Интерфейс больше не помогает игроку, а провоцирует его.

Примеры:

- HUD меняет цель на слишком личную;
- подсказка `E — взаимодействовать` превращается в `E — согласиться`;
- окно сообщения закрывается, но оставляет слово на экране;
- меню паузы показывает не «ПАУЗА», а «ожидание ответа».

### False mundane explanation

Каждый первый страшный эпизод должен иметь бытовое объяснение:

- глюк лампы;
- уведомление из рабочего чата;
- зависший смарт-замок;
- баг в UI;
- перегрев монитора.

Только второй или третий повтор должен разрушать объяснение.

## Правила текста

Текст должен быть коротким и сухим.

Плохо:

```text
Я вижу тебя, Руслан. Я теперь в твоём доме.
```

Лучше:

```text
Комната обнаружена.
```

Плохо:

```text
Ты умрёшь, если не отправишь коммит.
```

Лучше:

```text
merge required before sleep.
```

## Референсные ощущения сцен

Не копировать сцены из референсов. Брать только принципы.

| Референс | Что брать | Что не брать |
|---|---|---|
| Fears to Fathom | бытовая тревога, сообщения, ожидание | конкретные сюжеты и сцены |
| Trash Horror | короткая странность, немного нелепый быт | пародийность вместо хоррора |
| PSX horror | грязный визуальный шум, простая геометрия | узнаваемые модели/музыку |
| Analog horror | сухие системные сообщения | длинные экспозиционные ролики |
| Рабочие баг-репорты | узнаваемые формулировки | реальные бренды и сервисы |

## Микродетали квартиры

Эти детали важнее большого окружения:

- кружка рядом с клавиатурой;
- стикер с задачами;
- роутер с мигающим индикатором;
- смарт-колонка, которая сначала кажется бесполезной;
- зарядка телефона;
- свет от монитора на стене;
- тихий вентилятор компьютера;
- дверь, у которой есть маленький зелёный индикатор;
- полка с вещами, которые не должны двигаться.

## Дизайн-правило для каждой новой сцены

Перед реализацией сцены нужно ответить:

1. Какая бытовая цель игрока?
2. Что в сцене выглядит нормальным?
3. Какое одно маленькое нарушение происходит?
4. Как игрок может это объяснить рационально?
5. Что ломает это объяснение?
6. Какой звук является главным?
7. Как сцена меняет следующую цель?

Если ответы не ясны, сцену рано кодить.
