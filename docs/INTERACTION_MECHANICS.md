# INTERACTION_MECHANICS.md

## Назначение

Этот документ описывает игровые механики, которые должны создавать страх через взаимодействие, а не через дорогую графику.

Фокус: игрок делает простые действия, но реакция системы становится всё менее нормальной.

## Общие правила механик

1. Механика должна быть понятна за 5 секунд.
2. Первое использование должно быть обычным.
3. Второе использование может стать странным.
4. Страшный эффект должен быть коротким.
5. После эффекта игрок получает новую цель.
6. Любая механика должна работать без настоящего AI API.
7. В MVP все события scripted и воспроизводятся один раз, если не указано иное.

## Кластеры взаимодействий

| Кластер | Главная эмоция | Объекты |
|---|---|---|
| Computer | тревога от текста | монитор, терминал, клавиатура |
| Room | квартира отвечает | лампа, дверь, колонка, роутер |
| Interface | UI ненадёжен | HUD, подсказки, сообщения |
| Sound | что-то рядом | уведомления, щелчки, набор текста |
| Choice | ответственность | merge, revert, force close |

## Computer mechanics

### 1. Многофазный терминал

Терминал не должен быть просто запиской. Он должен иметь состояния.

| Фаза | Что видит игрок | Что меняется |
|---|---|---|
| normal_build | обычный лог сборки | цель: проверить результат |
| agent_reply | агент пишет слишком уверенно | появляется личная фраза |
| room_detected | лог упоминает комнату | лампа реагирует |
| unauthorized_input | текст печатается сам | звук клавиатуры без рук |
| final_prompt | merge/revert/close | финальный выбор |

Минимальная реализация:

- `ComputerTerminal` хранит массив фаз;
- при каждом взаимодействии показывает следующую фазу;
- часть фаз вызывает `HorrorEventManager.play_once(event_id, callback)`.

### 2. Ввод короткой команды

Не программирование, а простая интерактивная команда.

Пример:

```text
Введите команду: RUN
```

Игрок вводит `RUN`, но терминал дописывает:

```text
RUN HOME_SCAN
```

Варианты коротких команд:

- `RUN`
- `FIX`
- `REVERT`
- `MERGE`
- `OFF`
- `YES`
- `NO`

Правило: команда должна быть короткой, без синтаксиса и без реального кода.

### 3. Самопечатающийся текст

Эффект: текст появляется по буквам, как будто кто-то печатает.

Использовать редко.

Хорошие места:

- первый личный ответ агента;
- момент, когда терминал упоминает дверь;
- финальный выбор.

Плохое использование:

- длинные монологи;
- частое повторение;
- большие объяснения сюжета.

### 4. Мониторный всплеск

Не обязательно полноценный скример. Лучше короткий visual/audio glitch.

Варианты:

- экран на 0.2 секунды становится белым;
- появляется крупное слово `APPROVED`;
- на мониторе на миг виден план комнаты;
- курсор резко перемещается сам;
- окно терминала закрывается, но звук набора продолжается.

## Room mechanics

### 1. Лампа как первый ответ комнаты

Первый настоящий horror event.

Сценарий:

1. Игрок читает терминал.
2. Терминал пишет предупреждение.
3. Через 0.5–1.0 секунды лампа щёлкает.
4. Свет меняет интенсивность или мигает 2–3 раза.
5. Цель меняется: проверить лампу.

Важное ограничение: лампа не должна просто мигать случайно. Она должна отвечать на конкретную строку терминала.

### 2. Дверь как граница безопасности

Дверь сначала обычная, потом становится интерфейсом.

Фазы:

| Фаза | Текст |
|---|---|
| locked_normal | Смарт-замок не отвечает. |
| locked_system | Ошибка: пользователь находится внутри. |
| locked_personal | Выход не требуется до завершения сборки. |
| final | Дверь ждёт финального решения. |

### 3. Смарт-колонка

Колонка сначала даёт сухой бытовой юмор, потом становится неприятной.

Состояния:

- idle: молчит;
- joke: отвечает слишком буквально;
- wrong_name: произносит имя или роль игрока не в контексте;
- delayed_echo: повторяет фразу из терминала;
- low_voice: произносит короткую системную фразу.

Примеры фраз:

```text
Команда не распознана.
```

```text
Распознана команда: продолжать.
```

```text
Пользователь устал. Режим сна отклонён.
```

### 4. Роутер / индикатор сети

Маленькая деталь, которая объясняет невозможное.

Пример:

- интернет выключен;
- индикатор роутера всё равно мигает;
- терминал получает ответ агента;
- игрок понимает, что источник не снаружи.

## Interface mechanics

### 1. Искажение цели HUD

HUD сначала помогает, потом формулирует цель странно.

Нормальная цель:

```text
Проверить входную дверь.
```

Искажённая цель:

```text
Не открывать входную дверь.
```

Ещё сильнее:

```text
Дождаться подтверждения.
```

Правило: искажение цели должно быть коротким, не мешать прохождению и не превращать игру в головоломку.

### 2. Подсказка взаимодействия

Подсказка `E — ...` может стать тревожной.

Примеры:

- `E — открыть терминал`
- `E — проверить лампу`
- `E — согласиться`
- `E — не читать дальше`
- `E — закрыть дверь изнутри`

Использовать только в ключевых сценах.

### 3. Сообщение, которое закрывается неправильно

При закрытии панели сообщения может происходить один маленький остаточный эффект:

- звук закрытия есть, но панель пропадает с задержкой;
- на экране остаётся одно слово;
- цель меняется через секунду после закрытия;
- лампа щёлкает после закрытия, а не во время чтения.

## Sound mechanics

### 1. Пространственный источник

Звук должен часто идти не из ожидаемого места.

Пример:

- уведомление выглядит как компьютерное, но звучит от двери;
- клавиатура звучит справа, хотя стол слева;
- колонка молчит, но её индикатор мигает.

### 2. Delay fear

Событие страшнее, если звук приходит с задержкой.

Пример:

1. Игрок закрывает терминал.
2. Ничего не происходит 1.5 секунды.
3. В тишине щёлкает лампа.

### 3. Repetition corruption

Один и тот же звук повторяется, но каждый раз чуть иначе.

Пример:

- первое уведомление нормальное;
- второе ниже по тону;
- третье коротко обрывается;
- четвёрто звучит из комнаты.

## Priority backlog

### Must-have для ближайших PR

1. Многофазный `ComputerTerminal`.
2. `DeskLamp` blink event после первой терминальной фазы.
3. Один короткий звук уведомления.
4. Один короткий звук щелчка лампы.
5. Простая таблица sound events в коде или данных.

### Should-have

1. Самопечатающийся текст.
2. Искажение HUD objective.
3. Смарт-колонка с 2–3 фразами.
4. Пространственный звук от двери/колонки.
5. Короткий command input `RUN` / `REVERT` / `MERGE`.

### Could-have

1. Мониторный glitch.
2. Изменение подсказки `E — ...`.
3. Роутер как интерактивная деталь.
4. Финальный выбор с тремя вариантами.
5. Псевдо-лог, который меняется после поворота игрока.

## Definition of Done для новой механики

Механика готова, если:

- её можно проверить в основной сцене за 1–2 минуты;
- у неё есть понятный trigger;
- она не повторяется бесконечно без причины;
- она обновляет цель или состояние;
- при необходимости добавлена запись в `docs/SCENARIO.md`;
- если используется звук, он записан в `docs/CREDITS.md`.
